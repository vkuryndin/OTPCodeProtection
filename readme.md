# OTP Service

Сервис для защиты операций одноразовыми OTP-кодами.  
Поддерживает регистрацию и вход пользователей, разграничение ролей `ADMIN` и `USER`, настройку параметров OTP, генерацию и проверку кодов, а также доставку по нескольким каналам.

## Что умеет сервис

- регистрация пользователей;
- вход и выход по токену;
- разграничение доступа для `ADMIN` и `USER`;
- настройка длины OTP и времени жизни кода;
- генерация OTP для `operationId`;
- проверка OTP;
- автоматический перевод просроченных кодов в статус `EXPIRED`;
- доставка OTP через:
    - файл;
    - email;
    - Telegram;
    - SMS через SMPP-эмулятор.

## Основные сущности

- `users` — пользователи системы;
- `otp_config` — конфигурация OTP;
- `otp_codes` — сгенерированные коды.

Статусы OTP:
- `ACTIVE`
- `USED`
- `EXPIRED`

## Роли

### ADMIN
Администратор может:
- изменять конфигурацию OTP;
- получать список обычных пользователей;
- удалять пользователей и их OTP-коды.

### USER
Обычный пользователь может:
- генерировать OTP для операции;
- проверять OTP;
- привязывать Telegram для получения кодов.

## Каналы доставки

### FILE
Код сохраняется в указанный файл.

### EMAIL
Код отправляется на email пользователя, сохраненный в базе.

### TELEGRAM
Для Telegram используется отдельный сценарий привязки:
1. пользователь логинится в системе;
2. запускает привязку Telegram;
3. получает письмо со ссылкой на бота;
4. открывает ссылку и нажимает `Start` в Telegram;
5. сервис сохраняет `telegram_chat_id`;
6. после этого OTP можно получать в Telegram автоматически.

### SMS
Для SMS используется SMPP-эмулятор.  
В учебном режиме сервис работает с локальным SMPP simulator, реальный номер телефона не требуется.

## Конфигурация

Для запуска нужно настроить:
- подключение к PostgreSQL;
- JWT secret и срок жизни токена;
- SMTP для email;
- параметры Telegram-бота;
- параметры SMPP simulator.

Также для тестов используется отдельная тестовая база данных.

## Проверка работы

Основные сценарии:
- регистрация пользователя;
- вход в систему;
- генерация OTP;
- проверка OTP;
- изменение `otp_config` от имени администратора;
- привязка Telegram;
- доставка OTP через доступные каналы.

## Тестирование

В проекте есть:
- unit-тесты для основной бизнес-логики;
- интеграционные API-тесты с отдельной тестовой базой.

Интеграционные тесты проверяют:
- health endpoints;
- регистрацию и логин;
- logout и инвалидирование токена;
- генерацию и проверку OTP;
- валидацию запросов;
- admin endpoints;
- каскадное удаление OTP при удалении пользователя;
- сценарии привязки Telegram.

## Особенности реализации

- используется PostgreSQL 17 и JDBC;
- приложение собрано на Spring Boot;
- токены инвалидируются после logout;
- в таблице `otp_config` допускается только одна запись;
- при удалении пользователя связанные OTP-коды удаляются автоматически;
- логирование ведется в консоль и в файл.

## Назначение проекта

Проект реализован как учебный backend-сервис для демонстрации:
- работы с авторизацией и ролями;
- защищенных операций через OTP;
- интеграции с несколькими каналами доставки;
- тестирования REST API и бизнес-логики.
